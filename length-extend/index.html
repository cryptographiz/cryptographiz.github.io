Prerequisites

A cryptographic hash function is a one-way function that takes an arbitrary length input, and converts it to a fixed size output, called a digest or hash. Let hhh denote a good cryptographic hash function, then it should have the following properties:

  • Be deterministic. Given the same message, it always outputs the same digest.
  • A small change to a message must lead to entirely, uncorrelated, new digest.
  • Collision resistance: We should not be able to easily find two different messages such that they generate the same digest: h(m1)=h(m2)h(m1) = h(m2)h(m1)=h(m2). Note that given the nature of the hash function, these messages must exists. There is only a finite number of outputs possible, while it can have any length message. The goal is to make it very difficult to find such messages.
  • It is resistant against second pre-image attacks: given the message mmm we cannot easily find another m′m'm′ such that h(m)=h(m′)h(m) = h(m')h(m)=h(m′)
  • It is resistant against pre-image attacks. Given the digest ddd, one cannot easily find a message mmm such that h(m)=dh(m) = dh(m)=d.

Applications

Cryptographic hash functions are not just for checking integrity of a file after downloading.

They are also used for password verification. Instead of storing passwords in a database, the hash of a password is stored. That way, leaking the hash does not automatically leak the password. (It is slightly more complicated, see OWASP cheat sheet for password storage)

Other usages are in digital signatures or as proof of work for blockchains.

The usage we are interested in is that of a Message Authentication Code (MAC).

Attack

Given a digest of a secret key appended with a message, H(K∣∣M)=dH(K || M) = dH(K∣∣M)=d. One can generate a new hash of a new message appended to the old message, without knowing the secret key. H(K∣∣M∣∣pad∣∣M2)=d′H(K || M || pad|| M2) = d'H(K∣∣M∣∣pad∣∣M2)=d′.

The following algorithms are vulnerable:

  • MD4
  • MD5
  • SHA-0
  • SHA-1
  • SHA256 and SHA512(only non truncated versions of SHA2)
  • RIPEMD-160
  • WHIRLPOOL

All algorithms above are based on a Merkle-Damgård construction.

First, the message is split into chunks. In the example above this is 5 characters, for SHA-256 and MD5 it is 64 bytes total for one chunk. After that, the message is padded. Some algorithms also use the length of the message in the padding. For example, SHA-256 pads until it is a multiple of 56 bytes, after which the last 8 bytes are used to encode the length of the message, often in bits. Note that there is a different in endianess of the padding! SHA family uses little endian, whereas MD5 uses big endian.

The original message, including padding will then be used to output the digest. We can extend said digest without knowing the actual value of the original message! We simply take the digest as state and add new data by hashing it further.

When generating a new message to append, we need to know the length of the key and the message, to make sure our padding matches. This if often not known, luckily we can simply try a few different lengths and see if it works.

Example

Let's say we get the following digest: 5ed55f4f45eae63b83b0da2876997cfdbe45280587a69e29f2d5553888c62e72, which is the result of SHA256(secret || "Admin=false"). Suppose we didnt know the "secret", but only that it is 6 characters long and that the original data is "Admin=false".

We can still calculate the hash with the append data! For example, we want to add: "&viewFlag=True". The new digest then is 334d39883fb471d657d3bd8779bbc2659bd1e07964e20a8f06e3bd12490c12b1.

Note however, that the extra data is not directly appended, but first the padding is included. This is because the original digest is not just the result of the message, but also the padding that came after the message. If we extend with new data, we do that after the previous padding. Defense

One should not use hash functions that are vulnerable to length extension attack directly as MAC. Instead, one can use HMAC
